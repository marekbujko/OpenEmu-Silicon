# Libretro architecture in OpenEmu-Silicon

How a user-installed RetroArch core ends up running inside `OpenEmu.app`, and where in the tree to make changes that affect every libretro core at once.

If anything here drifts from the code, the code wins — fix this doc in the same PR.

---

## TL;DR

A `.dylib` from the user's RetroArch install on disk → an `.oecoreplugin` wrapper generated by `PrefCoresController` → loaded by `OELibretroCoreTranslator` (the **libretro host**) → presented to the rest of the app as an `OEGameCore` → run inside `OpenEmuHelperApp`. There are no libretro cores in this repo; the host runtime exists to load externally-built ones.

---

## How a RetroArch core gets into the app

The picker pipeline in `OpenEmu/PrefCoresController.swift`:

1. **Scan.** `scanRetroArchCores()` enumerates `~/Library/Application Support/RetroArch/cores/`.
2. **Parse metadata.** For each `.dylib`, the matching `<name>.info` file (next to it on disk) is parsed for display name, supported systems, and capability flags.
3. **Wrap.** When the user picks a core for a system, `installRetroArchPlugin(_:)` generates a synthetic `.oecoreplugin` bundle whose `Info.plist` points to the chosen `.dylib` via the `OELibretroCorePath` key. The bundle is named `<CoreName>-RetroArch.oecoreplugin` (see `RetroArchCore.pluginName`).
4. **Register.** The wrapper lands in the standard cores directory (`~/Library/Application Support/OpenEmu/Cores/`) and from then on is treated like any other core plugin.

The user-facing label is **"RetroArch core"** — that's what they see in Preferences. Internally the abstract concept (any `.dylib` implementing the libretro ABI) is a **libretro core**.

---

## The `.oecoreplugin` wrapper

It's a normal macOS bundle. The relevant Info.plist fields:

| Key | Purpose |
|---|---|
| `CFBundleIdentifier`, `CFBundleName` | Standard bundle metadata |
| `OEGameCorePluginClassName` | Points to the libretro host class |
| `OELibretroCorePath` | Absolute path to the user's external `.dylib` |
| `OEGameCoreSystemIdentifiers` | Which OpenEmu systems this core handles |

The host app's plugin loader doesn't care that the binary is wrapped — it loads the bundle, instantiates the plugin class, and the plugin class is the libretro host configured to `dlopen` the `.dylib` at the recorded path.

## The libretro host (`OELibretroCoreTranslator`)

`OpenEmu-SDK/OpenEmuBase/OELibretroCoreTranslator.m` (~1,990 LOC). Subclasses `OEGameCore` and implements the libretro ABI on the inside, OpenEmu's renderer/input/audio interfaces on the outside. Key entry points:

| Method | Line (approx.) | Role |
|---|---|---|
| `-loadFileAtPath:error:` | 982 | `dlopen` the libretro `.dylib`, call `retro_init` / `retro_load_game`, query memory and AV info |
| `-executeFrame` | 1192 | One `retro_run` tick + AV/input marshalling |
| `-stopEmulation` | 1179 | `retro_unload_game` / `retro_deinit`, tear down |
| Env callback dispatch | (internal) | Handles `RETRO_ENVIRONMENT_*` requests from the core |

Cross-cutting libretro features go here, **once**. That is the entire reason the host exists.

## What the host handles today, and what it does not

**Handles:**
- libretro ABI translation (`retro_run`, AV callbacks, input device descriptors)
- audio/video bridging into OpenEmu's renderer + audio queue
- input bridging (controller mapping, keyboard, hotplug)
- save state passthrough via `retro_serialize` / `retro_unserialize`

**Does not handle yet:**
- **RetroAchievements.** No `rc_client` initialization, no per-frame `rc_client_do_frame`, no memory exposure to rcheevos. Tracked in [#360](https://github.com/nickybmon/OpenEmu-Silicon/issues/360). Native cores integrate rcheevos directly; libretro cores get nothing.
- **`RETRO_ENVIRONMENT_SET_MEMORY_MAPS` (env callback 36).** Cores that expose RAM via descriptor tables rather than `retro_get_memory_data` are not currently mapped — verify before doing memory-dependent work (cheats, RA).
- **Runahead.** No frontend-side state-rollback loop.

---

## When to reach for a native core instead

Rule of thumb: if the user can install a working RetroArch core for the system, the host runtime is the right path. Reserve native cores (subclassing `OEGameCore` directly) for systems where in-tree control of the frame loop is necessary — currently RetroAchievements, anti-cheat behavior, and OE-specific features the libretro ABI doesn't expose. Adding a new in-tree libretro core is almost never the right answer; extend the host instead.

## Historical: bridge cores

Earlier in development the tree contained `Flycast-Bridge/`, `Gambatte-Bridge/`, and `VICE-Bridge/` directories holding pre-compiled libretro binaries used to test the host runtime. They were never shipped to users, never referenced by the Xcode project, and were removed in May 2026. They are mentioned here because older PRs and commits still refer to them. The current pathway for any libretro work is the host runtime plus the RetroArch picker — do not reintroduce in-tree libretro binaries.
